## Lab 2 — Knowledge Engineering & Domain Grounding
**Rubric: Domain Grounding (4) + Metadata Quality (3) + Viva (3)**

### Run live
```bash
python ingest_data.py
```

### What to Say
> "Lab 2 builds the agent's **source memory** using a RAG pipeline. The raw data comes from 6 simulated internal documents in the `data/` folder — supplier profiles, audit reports, compliance matrices, performance rankings.

> The pipeline has 6 stages:
> 1. **Load** — reads all `.txt` files from `./data/`
> 2. **Clean** — strips system export headers (`===== EXPORT =====`), record delimiters (`<<RECORD>>`), `Generated:` lines, and excessive whitespace
> 3. **Semantic chunk** — splits on double newlines, keeping each full supplier profile as one chunk. A procurement analyst needs the complete profile — certifications, lead times, risk notes — in one retrieval, not split across fragments
> 4. **Metadata enrich** — attaches 5 searchable tags to every chunk: `doc_type`, `supplier_id`, `region`, `category`, `priority_level`
> 5. **Vectorize** — uses `all-MiniLM-L6-v2` sentence-transformers locally (no API key needed)
> 6. **Index** — persists 28 chunks into ChromaDB at `./chroma_db/`

> The `retrieval_test.md` shows 3 real queries:
> - **Semantic search** — finds the correct backup semiconductor supplier (ALT-003) as top result
> - **category=audit filter** — restricts results to verified QA findings, surfaces RAW-008's CONDITIONAL PASS
> - **region=Europe filter** — returns only European suppliers, excluding all Asia-Pacific sources during a regional disruption scenario

> The metadata filtering is the key demonstration — without it, the agent can't distinguish audit findings from risk notes, or European suppliers from Asian ones."

## Lab 3 — The Reasoning Loop (ReAct with LangGraph)
**Rubric: Requirements (3) + Working (3) + Viva (4)**

### Run live
```bash
python graph.py
```

### What to Say
> "Lab 3 implements the **ReAct loop** — Reason + Act — using LangGraph. The graph has:
> - **State** — a `TypedDict` with a `messages` list using LangGraph's `add_messages` reducer. This appends messages rather than overwriting, preserving the full thought history
> - **Agent Node** — prepends a system prompt and calls Groq's `llama-3.3-70b-versatile` with 10 bound tools
> - **Tool Node** — `ToolNode(ALL_TOOLS)` from `langgraph.prebuilt` executes whatever tool calls the LLM generates
> - **Conditional Router** — checks if the last message has `tool_calls`. If yes → route to tools. If no → route to END

> Every tool uses the `@tool` decorator from `langchain_core.tools` and a Pydantic `BaseModel` input schema. This prevents the LLM from passing malformed arguments — if the schema says `top_k: int`, the LLM cannot pass a string.

> You'll see the agent call 6–8 tools in parallel, get results, reason about them, and produce a final response plan — all autonomously."

### Point out in output
- `[AI] Tool calls:` lines → LLM decided which tools to call
- `[Tool]:` lines → results flowing back
- Final `[AI]:` block → synthesized response

### Key Files
| File | Purpose |
|------|---------|
| `tools.py` | 10 tools with `@tool` + Pydantic — `ALL_TOOLS` list at bottom |
| `graph.py` | StateGraph, Agent Node, Tool Node, Router |

## Lab 4 — Multi-Agent Orchestration
**Rubric: Requirements (4) + Working (3) + Viva (3)**

### Run live
```bash
python multi_agent_graph.py
```
*(or just show `collaboration_trace.log` if time is short)*

### What to Say
> "Lab 4 evolves the single agent into a **team of specialists**. The motivation is 'instruction creep' — as a single agent's task grows complex, it starts forgetting earlier rules. Splitting into specialists fixes this.

> I have two agents:
> - **ResearcherAgent** — has only the 7 read-only and calculation tools. Its job is to gather raw facts. It cannot draft plans or send notifications.
> - **AnalystAgent** — has only the 3 action tools: `draft_response_plan`, `send_notification`, `update_purchase_order`. It synthesizes the Researcher's findings into decisions.

> The **handoff protocol** is a signal string: when the Researcher's work is complete, it ends its final message with `[HANDOFF: Research complete. Passing to Analyst.]`. The LangGraph router detects this in `route_researcher()` and transitions to the Analyst node.

> The graph has **4 nodes**: `researcher`, `researcher_tools`, `analyst`, `analyst_tools`. Each tool node is scoped — the Researcher's tool node only knows about researcher tools, so even if the LLM tried to call `send_notification`, it wouldn't be in scope."

### Point out in output / trace
- `[RESEARCHER — Tool Calls]` → Researcher gathering data
- `[RESEARCHER → HANDOFF — Response]` → handoff signal detected
- `[ANALYST — Tool Calls]` → Analyst drafting plan + notifying

## Lab 5 — State Management & Human-in-the-Loop
**Rubric: Requirements (4) + Working (3) + Viva (3)**

### Run live (Task 1: Persistence)
```bash
python persistence_test.py
```

### What to Say — Persistence
> "Lab 5 Task 1 solves the **session amnesia problem** — previously, stopping the Python script wiped the agent's memory. Using `SqliteSaver`, I compile the graph with a checkpointer:

> ```python
> graph.compile(checkpointer=SqliteSaver.from_conn_string('checkpoint_db.sqlite'))
> ```

> Every time the graph runs a step, it saves the full State to `checkpoint_db.sqlite`. The key is the `thread_id` — passing the same thread_id in the config config causes LangGraph to automatically reload the prior state before running.

> Watch the output: Turn 1 asks about alternative suppliers. Turn 2 asks 'what are their pricing differences?' — the word 'their' only makes sense if the agent remembers Turn 1. It does. The checkpoint shows 19 messages after Turn 1, 26 after Turn 2."

### Run live (Tasks 2 & 3: HITL)
```bash
python approval_logic.py
```

### What to Say — HITL
> "Lab 5 Tasks 2 and 3 implement **safety breakpoints**. The graph is compiled with `interrupt_before=['tools']`. This causes LangGraph to pause before executing ANY tool call, save the state to SQLite, and return control to my Python code.

> My code then inspects the pending tool calls. If the tool is `send_notification` or `update_purchase_order` — world-changing tools — it triggers the approval gate. Watch the `[!] SAFETY BREAKPOINT` banner appear.

> Task 3 demonstrates **state editing**: for the `send_notification` call, the human modifies the message body using `graph.update_state()` to inject an edited `AIMessage` with new tool call arguments. The graph then resumes with `graph.invoke(None, config)` and sends the human-edited version — not the original.

> The `checkpoint_db.sqlite` file is the physical proof that state is persisted between invocations."

### Show
- `checkpoint_db.sqlite` exists in the directory
- Point to the "PROOF OF PERSISTENCE" section in `persistence_test.py` output
- Point to `[Human edited]` line in `approval_logic.py` output

## If Asked "Why did you use X?"

| Choice | Answer |
|--------|--------|
| **Groq** | Free tier, fastest inference for llama-3.3-70b, no cold starts |
| **ChromaDB** | Local persistence, no API key, metadata filtering built-in, Python-native |
| **all-MiniLM-L6-v2** | Small (22M params), runs locally, well-benchmarked for semantic similarity |
| **LangGraph over CrewAI/AutoGen** | Explicit graph control, deterministic routing, production-grade checkpointing |
| **SqliteSaver over MemorySaver** | Persists to disk — proves cross-session recovery, not just in-process memory |
| **5 metadata tags** | Doc_type + supplier_id enables compound filtering; region + category enable geographic and domain scoping; priority_level enables urgency triage |
